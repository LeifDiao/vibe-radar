# VibeRadar — 方法论

> VibeRadar 关注的不是代码产出本身，而是你如何与 AI 建立高质量协作。
>
> 这份文档既是 Claude 打分时遵循的公开规则，也是报告中每项分数的解释依据。你可以用它理解评分逻辑，并按团队实际协作方式做相应调整。

---

## 1. 为什么要做 VibeRadar

你已经和 Claude Code 进行过很多次对话：写功能、改 bug、做重构。有些会话推进顺畅、决策清晰；有些则值得回头复盘，看看哪些协作环节还可以进一步优化。

VibeRadar 聚焦最影响协作质量的关键动作：上下文组织、指令表达、过程纠偏与结果验证。

它把这些原本分散的协作习惯整理成结构化报告，帮助你识别已有优势、发现改进空间，并持续优化与 AI 的配合方式。

大多数开发者工具衡量的是**产出**——代码行数、测试覆盖率、PR 吞吐量。VibeRadar 衡量的是**协作输入**：你怎么提问、怎么推进、怎么验证、怎么收尾。这些环节决定了 AI 协作最终带来的是更高的推进效率，还是额外的沟通与返工成本。

---

## 2. 设计原则

1. **证据优先。** 每个评分都要能回到具体、可计数的会话信号。
2. **隐私不妥协。** 数据留在本地，不上云、不传输、不采集。
3. **数据不足时保持克制。** 分数会进行置信度缩放，并附上明确提示，避免过度解读。
4. **建议必须可执行。** 每份报告都要给出 3 到 7 条能改变下一次会话的建议。
5. **轻量，才适合长期使用。** 插件不到 200 KB，零运行时依赖，输出单文件 HTML。
6. **位置决定含义。** 同一个信号出现在不同对话位置，不该被算成同一件事。
7. **公式保证一致性，Claude 补充语境判断。** 先有可复现的基线，再做有上限的上下文微调。

## 3. 工作原理

```
~/.claude/projects/<slug>/*.jsonl
         │
         ▼
   [parse-project.mjs]           ← 确定性。位置感知的信号提取。
         │ facts.json
         ▼
    [Claude in the skill]        ← 读取 rubric.json。两步评分：公式 + 微调。
         │ report.json
         ▼
   [render-report.mjs]           ← 纯转换。JSON + 模板 → HTML。
         │
         ▼
  ~/.vibe-radar/reports/<slug>-<ts>.html
```

三个彼此独立的阶段：

- **解析器**是确定性的。相同输入 → 相同事实，输出位置分类信号和 MBTI 行为信号。
- **评分器**是 Claude，运行在你的 Claude Code 会话中。它先根据信号计算**公式基线**，应用**置信度缩放**，再结合消息内容做有上限的定性微调（最多 ±15 分）。
- **渲染器**负责生成单文件 HTML 报告，数据、样式和 JS 全部内联。

整个流程没有外部 API、没有云端处理，也不依赖服务器。

## 4. 核心创新：位置感知信号

### 全局计数的问题

一种朴素的方法是全局统计信号——不管 `hasFilePath` 出现在哪里，都一视同仁。但同一个信号在对话的不同位置含义完全不同：

| 位置 | `hasFilePath` 的含义 |
|---|---|
| **开场**（前 2 条消息） | 上下文——为 AI 搭建舞台 |
| **指令**（新任务指令） | 意图——精确指定要修改什么 |
| **纠正**（AI 出错后） | 反馈——指出 AI 错在哪里 |
| **确认**（AI 输出后的简短确认） | 无显著意义 |

全局计数导致了**信号交叉污染**：一个经常引用文件路径的用户，会因为同一个行为在意图、上下文和反馈三个维度同时获得加分。一个信号影响了三个维度。

### 解决方案：将信号路由到对应位置

VibeRadar 在统计信号之前，先将每条用户消息分类到 5 个位置之一：

| 位置 | 定义 |
|---|---|
| `opening` | 每个会话的前 2 条用户消息 |
| `directing` | 新任务或指令（不是对 AI 输出的回应） |
| `correcting` | AI 产出输出后 + 用户表现出纠正意图 |
| `confirming` | AI 产出输出后 + 用户给出简短确认 |
| `continuing` | 其他所有情况 |

每个维度**只读取其指定位置的信号**：

| 维度 | 主要位置 | 读取内容 |
|---|---|---|
| **瞄准力** | `directing` | *任务指令中的*目标清晰度、约束条件、具体性 |
| **画面感** | `opening` | *会话开场中的*技术栈、文件路径、项目结构 |
| **拆弹术** | `global` | 需求过载、渐进式结构（天然跨位置） |
| **导航力** | `correcting` | *AI 出错后纠正中的*推理、具体性 |
| **鉴定术** | `confirming` + global | *确认中的*盲目接受 + 主动测试请求 |
| **节奏感** | `global` | 总结、里程碑、话题连贯性（天然跨位置） |

这使得各维度**真正正交**：同一个原始信号根据对话上下文被路由到不同维度。

## 5. 六大维度

### 维度 1 — 瞄准力

**衡量内容：** 你在*发出新指令时*表达需求的精确程度。

**重要性：** 目标清晰度越高，AI 产出越稳定。模糊的请求通常只会带来模糊的结果。

**位置：** `directing` — 你给 AI 布置新任务的消息（不是对输出的回应）。

**信号（来自 directing 位置）：**

| 信号 | 捕捉内容 |
|---|---|
| `hasExpectedBehavior` | 目标设定性语言 |
| `hasConstraint` | 边界和限制条件 |
| `isVague` | 模糊表达："帮我 / 修一下这个" |
| `hasIdentifier` | 对具体代码元素的引用 |

**"沉默专家"模式：**

一种常见的假阴性场景：经验丰富的用户写 `"change validateToken in src/auth/middleware.ts:45 to async, return Promise<TokenResult>"`。这类表达已经非常精确，但在 `hasExpectedBehavior` 或 `hasConstraint` 上可能没有关键词匹配。公式基线会给出中等评分，Claude 的 ±15 分微调可以将其识别为高意图并适度上调。

**评分锚点：**

| 分数 | 对应表现 |
|---|---|
| 0-20 | 模糊的一句话，没有目标 |
| 21-40 | 有一些目标但经常不清楚 |
| 41-60 | 大多数时候中等清晰 |
| 61-80 | 大多数请求都有清晰的目标 + 约束 |
| 81-100 | 持续呈现清晰目标、明确约束和预期结果 |

---

### 维度 2 — 画面感

**衡量内容：** 你在*会话开始时*给 AI 提供了多少背景信息。

**重要性：** AI 在每次新会话开始时都缺少背景信息。上下文交代得越完整，进入有效协作的速度越快。

**位置：** `opening` — 每个会话的前 2 条用户消息。

**信号（来自 opening 位置）：**

| 信号 | 捕捉内容 |
|---|---|
| `hasFilePath` | 会话开场中的具体文件引用 |
| `hasTechStack` | 会话开场中的技术栈提及 |
| `hasError` | 预先提供的错误上下文 |
| `firstMessage.avgLength` | 开场消息长度 |
| `firstMessage.sessionsWithTechStack` | 在第一条消息中提及技术栈的会话数 |

---

### 维度 3 — 拆弹术

**衡量内容：** 你是否将复杂任务拆解为 AI 能处理的大小。

**重要性：** AI 更擅长聚焦且范围明确的工作。在一条消息中同时提出多个改动，往往会显著提高误解和返工风险。

**位置：** `global` — 任务拆解跨所有位置衡量。

**信号：**

| 信号 | 捕捉内容 |
|---|---|
| `patterns.demandOverloads` | 300+ 字符消息且含 3+ 个动作动词 |
| `patterns.longUnstructured` | 500+ 字符消息且无结构 |
| `labelRatios.progressive` | 分步语言 |
| `labelRatios.checkpoint` | 阶段标记 |
| `stats.avgHumanMsgChars` | 平均长度（>300 暗示批量堆砌） |

---

### 维度 4 — 导航力

**衡量内容：** 当 AI *偏离航向*时，你纠正得有多好。

**重要性：** AI 协作需要及时校正。高质量的纠偏能更快恢复方向，也能减少不必要的反复往返。

**位置：** `correcting` — 你对 AI 错误输出做出回应的消息。

**信号（来自 correcting 位置）：**

| 信号 | 捕捉内容 |
|---|---|
| `hasReasoning` | 解释*为什么*错的纠正 |
| `hasFilePath` | 引用确切文件位置的纠正 |
| `hasIdentifier` | 指向具体代码元素的纠正 |
| `patterns.retryLoops` | 3+ 条类似消息（反馈没有起效） |
| `patterns.noReplyToQuestion` | 忽视 AI 的澄清问题 |

**为什么位置很重要：** 如果没有位置感知，开场消息中的 `hasReasoning`（"我做这个是因为……"）会虚增反馈维度的分数。位置感知计数确保只有 correcting 位置的 `hasReasoning` 计入反馈。

---

### 维度 5 — 鉴定术

**衡量内容：** 你是否在信任 AI 输出之前先验证它。

**重要性：** AI 输出并不总是可靠。缺少验证，问题就更容易被带入后续工作。

**位置：** `confirming`（检测盲目接受） + `global`（主动测试）。

**信号：**

| 信号 | 捕捉内容 |
|---|---|
| `patterns.blindAccepts` | AI 产出大量输出后的简短确认（配合改进的误报过滤） |
| `labelRatios.requestTest` | 要求 AI 运行测试 |
| `labelRatios.thinkFirst` | 要求 AI 先解释再写代码 |
| `labelRatios.proactiveReview` | 主动审查 AI 的方案 |

**智能盲目接受检测：** 如果用户在前 3 轮内给出了清晰、具体的指令，那么 AI 输出后的简短"ok"**不会**被计为盲目接受。这可以减少这类误报：用户说"把所有 console.log 替换成 logger.info" → AI 执行 → 用户说"好的，下一步"——这属于对执行结果的确认，而不是缺少验证。

---

### 维度 6 — 节奏感

**衡量内容：** 你在长会话中对*对话节奏*的把控能力。

**重要性：** 漫无目的的会话会留下半成品功能和上下文负债。

**位置：** `global` — 会话流程天然是跨位置的。

**信号：**

| 信号 | 捕捉内容 |
|---|---|
| `patterns.topicDrifts` | 3+ 条连续不相关的消息（改进：忽略共享文件路径前缀的消息） |
| `labelRatios.summary` | 明确的总结/回顾 |
| `labelRatios.milestone` | 里程碑/进度语言 |
| `stats.validSessions` | 持续使用 |

**该维度的适用边界：** 主要处理短任务（bug 修复、小功能）的用户，天然较少使用里程碑和总结。在这类场景中，较低的 Architecture 分数并不代表协作质量较弱；如果它是明显的短板维度，报告的洞察应当说明这一点。

## 6. 评分机制：公式 + 微调

### 为什么不只用公式？

关键词匹配有盲区。纯公式很难识别：一条 70 字符、带文件路径和方法名的消息，可能比一条 300 字符、充满"我希望"和"应该"的消息更精确。Claude 可以补足这部分语境理解。

### 为什么不完全交给 Claude？

LLM 评分存在约 ±5 分的运行间方差。相同数据多次评分可能得到不同结果，这会削弱评分的一致性和可信度。

### 兼顾稳定性与语境的评分方式

1. **公式基线**（确定性）：相同数据 → 相同分数，永远如此。
2. **置信度缩放**：数据稀疏时，分数向 50 收缩。
3. **Claude 微调**（有上限）：最多 ±15 分，必须引用证据。

这样既能把运行间方差控制在约 ±3 分，又能保留对语境的理解。

### 基线公式设计

每个维度在 `rubric.json` 中都有一个公式。模式如下：

```
baseline = 50 + Σ (signal_ratio - midpoint) × weight
```

其中：
- `50` 是中性中心点
- `midpoint` 是"普通"用户的预期比率（根据早期测试校准）
- `weight` 决定每个信号对分数的影响幅度
- 结果被截断到 [0, 100]

中点和权重是透明且可调整的。你可以直接编辑 `rubric.json`，让评分标准更贴合自己的协作方式。

## 7. MBTI：从词汇到行为

### 基于关键词的 MBTI 的问题

纯粹基于词汇的方法是：说了"架构" → +1 N。说了"性能" → +1 T。但协作风格取决于*你做了什么*，而不是*你用了什么词*。

### 行为信号

每个轴现在以**行为模式**为主要信号，关键词为辅助：

#### E/I — 思维暴露度

| | E（外向型） | I（内向型） |
|---|---|---|
| **行为特征** | 消息包含推理链（>80 字符中有 2+ 个因果连接词） | 消息是简短指令（<120 字符且带 filePath/identifier） |
| **指标** | `reasoningChainRatio` | `directiveRatio` |
| **含义** | 在对话中展开思考，暴露推理过程 | 在内部思考完毕，只分享结论 |

#### S/N — 引用粒度

| | S（具象型） | N（直觉型） |
|---|---|---|
| **行为特征** | 消息频繁同时引用 filePath + identifier | 消息使用架构/设计语言但没有具体文件引用 |
| **指标** | `concreteRefDensity` | `abstractRefRatio` |
| **含义** | 一切落实到具体代码 | 以模式和系统思维来思考 |

#### T/F — 纠正优先级

| | T（逻辑型） | F（体验型） |
|---|---|---|
| **行为特征** | 纠正时引用性能、边界情况、正确性 | 纠正时引用可读性、用户体验、命名、美观 |
| **指标** | `correctingTCount` | `correctingFCount` |
| **含义** | 以正确为优化目标 | 以可读为优化目标 |

关键创新：**T/F 仅在 correcting 位置衡量**。在开场消息中，"性能"可能是在描述项目需求，而不是个人偏好。在纠正中，它揭示的是用户评估 AI 输出时真正在意什么。

#### J/P — 会话弧线

| | J（结构型） | P（探索型） |
|---|---|---|
| **行为特征** | 会话呈线性话题推进（连续消息间有高平均 Jaccard 相似度） | 会话频繁改变方向（Jaccard 下降） |
| **指标** | `sessionArcLinearity` | `directionChangeCount` |
| **含义** | 先计划再执行 | 先探索再调整 |

### 平衡值

VibeRadar 不输出独立的 E/I/S/N/T/F/J/P 计数，而是为每个轴输出一个**平衡值**（-1 到 +1）：

- `balance.ei > 0.1` → E，`< -0.1` → I，之间 → 边缘
- 幅度决定 `strength`（0-100 置信度）

### MBTI 不是什么

这里的 MBTI 是**描述性的**，不是规定性的。INTJ 并不比 ESTP "更好"。它们代表的是不同的协作风格和取舍，价值在于帮助你理解自己稳定的偏好与常见短板。

## 8. 置信度缩放

### 为什么会话数量很重要

| 会话数 | `confidenceLevel` | 缩放 |
|---|---|---|
| < 5 | `low` | 分数向 50 收缩 30% |
| 5-14 | `medium` | 分数向 50 收缩 15% |
| 15+ | `high` | 不缩放 |

只有 3 个会话时，一个异常会话就能让比率波动 0.3。置信度缩放防止"第一周"的分析看起来像盖棺定论。

缩放公式：`adjusted = 50 + (baseline - 50) × factor`

这意味着：
- 基线 80 在低置信度下变为 50 + 30 × 0.7 = **71**
- 基线 30 在低置信度下变为 50 - 20 × 0.7 = **36**

极端分数被拉向中心。中等分数几乎不变。报告在置信度不高时会明确提示。

## 9. 建议生成逻辑

每份报告都会生成 **3 到 7 条改进建议**。数量由真实数据决定，而不是固定模板。

### 优先级分配

| 维度分数 | 优先级 | 是否必须？ |
|---|---|---|
| 0-54 (D/C) | `high` | 是 |
| 55-69 (B) | `medium` | 通常 |
| 70-84 (A) | `low` | 仅在有具体短板时 |
| 85-100 (S) | 无 | 跳过 |

### 质量标准

每条建议都需要满足：

1. **具体，而非笼统。** "在接受代码后运行 `npm test`"，而不是"多测试"。
2. **基于具体证据。** "你有 14 次盲目接受"，而不是"你接受得太随意"。
3. **一个会话内可执行。** "在每次会话结束时做一个回顾"，而不是"提升你的架构能力"。

## 10. 信号盲区与可见性边界

VibeRadar 会明确说明自己看不到的部分：

1. **不可见的验证。** 在 IDE 中审查完 AI 的 diff 再输入"ok"的用户，和盲目接受的用户，在我们看来一模一样。我们看不到对话之外的行为。

2. **沉默的专业性。** 像 `"make validateToken async in middleware.ts:45"` 这样的消息已经非常精准，但在"预期行为"或"约束"的关键词匹配中可能得分为零。微调机制可以部分补偿，但关键词匹配始终无法完整覆盖这类精准表达。

3. **来自 CLAUDE.md 的上下文。** 维护良好 CLAUDE.md 文件的用户给了 AI 持久上下文，而无需在消息中重复。他们的 Context 得分可能低估了他们实际的上下文建设行为。

4. **结对编程模式。** 与 AI 进行快速短消息协作的用户，和习惯先写详细需求说明的用户，在信号层面会呈现不同特征。两者并无绝对优劣，但评分模型会反映这种差异。

5. **语言限制。** 关键词信号适用于英语和中文。其他语言在语言类维度上会被低估。

---

## 11. 我们不衡量的东西

- **代码质量。** 那是 linter、测试和代码审查的事。
- **语言/框架能力。** 不在我们的范围内。
- **绝对生产力。** 我们无法告诉你这周是否产出更多了。
- **Bug 修复成功率。** 我们看到的是对话，不是结果。
- **安全态势。** 那是一个完全独立的领域。

我们衡量的是**协作过程中的对话行为**，不是最终结果。

## 12. 已知局限

1. **小样本意味着大误差。** 置信度缩放可以缓解但无法消除。少于 5 个会话会产出明确标记为低置信度的报告。

2. **评分存在运行间方差。** 典型 ±3 分（通过公式锚定保持低位）。MBTI 类型比单个维度分数更稳定。

3. **当前范围只覆盖 Claude Code 会话。** 这套评分体系是围绕长对话式协作建立的。

4. **评分标准是有立场的。** 我们把“好的 AI 协作”定义为目标导向、重视验证、具备对话意识。若你的团队有不同偏好，也可以直接在 `rubric.json` 中调整标准。

5. **位置分类是启发式的。** 被分类为"directing"的消息，实际上可能是一个没有明确纠正关键词的纠正。大多数分类是有效的，但边界情况依然存在。

## 13. 认识论声明

**我们不主张：**

- 这经过了科学验证
- INTJ 协作者比 ESTP "更好"
- 低分意味着你是个差劲的开发者
- 我们的评分是唯一有效的框架

**我们确认的是：**

- 这 6 个维度覆盖了完整的"提问 → 验证 → 闭环"循环
- 位置感知信号使各维度真正正交
- 公式 + 微调的评分机制是可复现且可审计的
- 报告提供的是**结构化反馈**，而非价值判断
- 认真阅读报告并优先落实前 3 条建议，通常会改善后续 AI 协作质量

VibeRadar 的价值在于帮助你持续校准协作方式，而不是把开发者做简单排序。

## 14. 如何修改评分

一切都编码在 `data/rubric.json` 中：

- **修改基线公式** — 编辑 `dimensions.<dim>.baselineFormula`
- **修改信号路由** — 编辑 `dimensions.<dim>.primaryPosition` 和 `dimensions.<dim>.signals`
- **修改微调指引** — 编辑 `dimensions.<dim>.adjustmentGuide`
- **修改等级阈值** — 编辑 `grades[*].range`
- **修改 MBTI 画像** — 编辑 `mbti.profiles`
- **修改置信度缩放** — 编辑 `scoring.confidenceScaling`

无需修改代码。Claude 每次运行都会读取 `rubric.json`。
